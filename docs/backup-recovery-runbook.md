# 백업/복구 Runbook (REL-004)

본 문서는 운영 장애/데이터 손상 상황에서 서비스를 안전하게 복구하기 위한 표준 절차입니다.

## 1) 목적과 범위
- 목적: 결제/구독/환불/정책 동의 핵심 데이터의 손실 리스크를 최소화한다.
- 범위:
  - Supabase(PostgreSQL/Auth)
  - `.data` 디렉터리 기반 파일 스토어(감사로그, dead-letter, retry, refunds, payments)

## 2) 역할과 책임
- Incident Commander (IC): 장애 총괄, 의사결정, 커뮤니케이션 승인
- Recovery Engineer (RE): 기술 복구 실행
- Scribe: 타임라인/조치 내역 기록
- Support Owner: 공지/CS 대응

## 3) 복구 목표
- RPO(허용 데이터 손실): 최대 24시간
- RTO(허용 복구 시간): 최대 2시간

## 4) 백업 정책

### 4.1 Supabase
- PITR(Point-in-time recovery) 활성화
- 일별 백업 상태 체크
- 주 1회 백업 상태 스크린샷/로그 보관

### 4.2 파일 스토어 (`.data/*`)
- 백업 대상:
  - `.data/audit-logs`
  - `.data/dead-letter`
  - `.data/subscription-retries`
  - `.data/refunds`
  - `.data/payments`
  - `.data/policy-consents.json`
- 권장:
  - 일 1회 tar.gz 압축 백업
  - 백업 보존 기간: 30일

## 5) 장애 등급
- Sev-1: 결제/환불/로그인 핵심 기능 중단
- Sev-2: 부분 기능 장애, 우회 가능
- Sev-3: 성능 저하, 기능은 동작

## 6) 표준 복구 절차 (Sev-1 기준)

### Step 0. 초기 선언 (0~10분)
1. IC 지정
2. 장애 티켓 생성
3. 사용자 공지 초안 준비

### Step 1. 원인 분류 (10~20분)
1. 앱 에러 로그/Sentry 확인
2. Supabase 상태 페이지 확인
3. 최근 배포/환경변수 변경 확인

### Step 2. 즉시 완화 (20~40분)
1. 신규 배포 중지
2. 결제 라우트 장애면 결제 진입 임시 차단 배너 적용
3. 웹훅 실패 증가 시 dead-letter 큐 적재 확인

### Step 3. 데이터 복구 (40~90분)
1. Supabase PITR 복구 필요 여부 판단
2. 파일 스토어 손상 시 최근 백업 아카이브 복원
3. 복구 후 무결성 점검:
   - 결제 승인 건수 vs 환불 건수
   - dead-letter -> replay 가능 여부
   - 정책 동의 데이터 조회 가능 여부

### Step 4. 서비스 검증 (90~110분)
1. 스모크 체크:
   - 로그인
   - 단건 결제
   - 구독 결제
   - 환불 API
   - 웹훅 수신
2. 핵심 테스트 명령 실행:
   - `npm run test`
   - `npm run lint`
   - `npm run build`

### Step 5. 정상화 공지 (110~120분)
1. 장애 원인/영향/복구 완료 시각 공유
2. 사후 회고 일정 생성

## 7) 장애 커뮤니케이션 템플릿

### 7.1 탐지 공지
- 제목: [장애공지] 결제 기능 장애 인지 및 조치 중
- 내용: 발생 시각, 영향 범위, 다음 업데이트 예정 시각

### 7.2 복구 공지
- 제목: [복구완료] 결제 기능 정상화 안내
- 내용: 복구 시각, 원인 요약, 재발방지 계획

## 8) 복구 후 사후조치
- 24시간 내 포스트모템 작성
- 재발 방지 액션 아이템 생성
- `docs/execution-log.md`에 복구 이벤트 기록
