# 바이브코더 대표 AI 툴 4가지 실전 사례 가이드

이 문서는 바이브코더가 많이 사용하는 대표 도구 4가지를 기준으로, 이 보일러플레이트 프로젝트에서 실제로 어떻게 작업 지시를 내리고 검증하는지 사례 중심으로 설명합니다.

## 공통 원칙 (모든 툴 동일)
- 요청 전에 `목표/범위/완료기준/테스트`를 먼저 고정한다.
- 한 번에 티켓 1개만 처리한다.
- 작업 종료 시 반드시 `npm run test && npm run lint && npm run build` 결과를 남긴다.
- 작업 후 `docs/execution-log.md`에 기록한다.

---

## 1) Cursor Agent (IDE 내 구현 중심)

### 언제 쓰나
- 현재 열려 있는 파일을 빠르게 수정할 때
- UI/라우트/컴포넌트 연결을 한 번에 바꿀 때

### 실전 사례
- 티켓: `BILL-005` (결제 실패 사용자 CTA 추가)
- 요청 예시:
```md
티켓: BILL-005
목표: 결제 실패 사용자에게만 결제수단 변경 CTA를 노출
범위: app/payments, components
제약: 기존 success/fail 라우트 구조 유지
DoD:
- 실패 사용자에게만 CTA 표시
- 성공 사용자/비로그인 미노출
검증:
- npm run test
- npm run lint
- npm run build
```

### 작업 후 체크
- 변경 파일 목록 확인
- 테스트 명령 결과 확인
- execution-log 기록

---

## 2) Claude Code (큰 단위 구조화/검증 루프)

### 언제 쓰나
- PRD/백로그/문서/테스트를 함께 정리할 때
- 구현 + 검증 + 문서화를 한 번에 끝낼 때

### 실전 사례
- 티켓: `WH-003` (웹훅 서명 검증)
- 요청 예시:
```md
WH-003만 진행해줘.
먼저 관련 파일 읽고 3줄 계획 제시 후 수정해.
서명 검증 실패는 401, 성공은 기존 흐름 유지.
테스트코드 추가하고 test/lint/build 실행 결과까지 보고해.
execution-log에 단계 기록까지 남겨줘.
```

### 작업 후 체크
- 설계 근거가 포함됐는지 확인
- 실패 시 원인/대안까지 보고됐는지 확인

---

## 3) GitHub Copilot Chat (VSCode) (빠른 코드 초안/리팩터링)

### 언제 쓰나
- 작은 함수 리팩터링이나 테스트 초안을 빨리 만들 때
- 특정 파일 범위에서 반복 코드를 정리할 때

### 실전 사례
- 티켓: `REL-003` (Rate Limit + 보안 헤더)
- 요청 예시:
```md
현재 파일 기준으로 rate limit 미들웨어 초안 생성해줘.
요구사항:
- /api/toss/* 경로만 적용
- 분당 요청 한도 초과 시 429
- 기존 로직 영향 최소화
추가로 테스트 케이스 3개 제안해줘.
```

### 작업 후 체크
- 초안은 반드시 사람이 검토 후 반영
- 보안 관련 코드는 테스트 없으면 병합 금지

---

## 4) Windsurf / Codeium Agent (대규모 코드 탐색/빠른 반복)

### 언제 쓰나
- 레거시/대형 프로젝트에서 관련 파일을 폭넓게 찾을 때
- 반복 수정과 빠른 비교를 자주 돌릴 때

### 실전 사례
- 티켓: `OPS-002` (부분 환불)
- 요청 예시:
```md
부분 환불과 관련된 파일/함수 후보를 먼저 전부 찾아줘.
그 다음 최소 변경으로 환불 누적 한도 검증을 구현해.
필수:
- 기존 전액 환불 로직 유지
- 회귀 테스트 추가
- test/lint/build 결과 포함
```

### 작업 후 체크
- "관련 파일 탐색 결과"를 먼저 받았는지 확인
- 최소 변경 원칙이 지켜졌는지 확인

---

## 도구 선택 기준 요약
- Cursor: 구현 속도/IDE 밀착 작업
- Claude Code: end-to-end 처리(기획+코드+검증+문서)
- Copilot Chat: 코드 초안/로컬 리팩터링 보조
- Windsurf/Codeium: 넓은 탐색 + 반복 수정 효율

---

## 실제 사용 순서 (권장)
1. Claude Code로 티켓 범위/DoD 확정
2. Cursor 또는 Windsurf로 구현
3. Copilot Chat으로 테스트 초안/리팩터링 보조
4. Claude Code로 최종 품질 점검 + 문서 정리
5. `execution-log` 기록

---

## 실패 확률 낮추는 운영 팁
- "한 번에 다 해줘" 대신 "티켓 1개만" 요청
- 항상 수정 범위를 파일 경로로 제한
- 검증 명령을 프롬프트에 명시
- 결과 보고 형식(변경파일/테스트결과/리스크)을 고정
