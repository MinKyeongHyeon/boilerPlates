# Claude Code / Cursor 실전 0 to 10 가이드

이 문서는 "키를 넣고 실행" 수준이 아니라, 비개발자가 실제로 기능을 안전하게 추가/수정/검증/배포하기 위한 운영 매뉴얼입니다.

---

## 0. 이 문서의 목표
- 목표 1: AI 툴에 일을 잘 시키는 방법을 익힌다.
- 목표 2: 망가뜨리지 않고 기능을 추가하는 루틴을 만든다.
- 목표 3: 실패했을 때 복구 가능한 방식으로 작업한다.

---

## 1. 시작 전에 꼭 이해할 원칙 5개
1. AI는 빠르지만, "요구사항이 모호하면" 잘못된 코드를 빠르게 만든다.
2. 한 번에 큰 요청보다, 작게 쪼개서 단계별로 시키는 것이 품질이 높다.
3. 항상 "완료 기준(DoD)"을 먼저 알려줘야 한다.
4. 코드 수정 후에는 반드시 `test/lint/build`로 검증한다.
5. 작업 로그를 남겨야 다음 작업이 빨라지고 안전해진다.

---

## 2. 작업 전 준비 (매번 동일)
터미널에서 아래 순서로 확인:

```bash
npm install
npm run test
npm run lint
npm run build
```

- 하나라도 실패하면 새 기능 작업을 시작하지 말고 실패 원인부터 해결한다.
- 모든 명령이 성공한 상태를 "기준선(Baseline)"으로 둔다.

---

## 3. Cursor에서 AI에게 요청하는 기본 포맷
아래 템플릿을 그대로 복사해서 쓰면 품질이 가장 안정적이다.

```md
[목표]
이번 작업의 목표는 무엇인가

[변경 범위]
수정 가능한 파일/수정 금지 파일

[완료 기준(DoD)]
- 항목1
- 항목2

[테스트 기준]
- 어떤 케이스를 통과해야 하는가

[출력 형식]
- 변경 파일 목록
- 실행한 검증 명령과 결과
- 남은 리스크
```

---

## 4. Claude Code에 기능 추가를 시킬 때의 실전 순서

### Step 1) 요구사항을 3줄로 고정
- 문제: 지금 어떤 문제가 있는지
- 목표: 어떤 상태가 되면 성공인지
- 제약: 변경하면 안 되는 것

예시:
- 문제: 정기결제 실패 시 사용자 안내가 없다.
- 목표: 실패 시 결제수단 변경 CTA를 보여준다.
- 제약: 기존 결제 성공/실패 라우트 구조는 유지한다.

### Step 2) 작업 단위를 티켓으로 분할
- 한 번에 티켓 1~2개만 진행
- 예: `BILL-005`만 먼저 완료

### Step 3) "먼저 읽고, 그다음 수정"을 명시
AI에게 아래를 반드시 같이 전달:
- "수정 전 관련 파일을 먼저 읽고 계획을 3줄로 출력해"
- "그 다음 코드 수정하고 test/lint/build까지 실행해"

### Step 4) 변경 후 결과를 강제 보고
- 변경 파일 목록
- 왜 이 방식으로 수정했는지
- 실패한 테스트가 있으면 원인/대안

---

## 5. 좋은 프롬프트 / 나쁜 프롬프트

### 나쁜 프롬프트
- "결제 좀 안정적으로 해줘"
- 문제/범위/완료기준이 없음

### 좋은 프롬프트
- "BILL-005 티켓만 진행해줘.
  변경 범위는 `app/payments`, `components`만 허용.
  완료 기준은 실패 결제 사용자에게만 CTA 노출, 비로그인/성공 유저 노출 금지.
  끝나면 `npm run test && npm run lint && npm run build` 결과까지 보고해."

---

## 6. 안전하게 반복 개발하는 루틴 (권장)
1. 백로그에서 티켓 1개 선택
2. AI에게 범위/DoD/테스트 전달
3. 코드 수정
4. `test/lint/build` 실행
5. `docs/execution-log.md`에 기록
6. 다음 티켓으로 이동

이 루틴을 깨지 않으면, 비개발자도 품질을 유지하면서 속도를 낼 수 있다.

## 7. 실제 운영에서 자주 터지는 문제와 요청 문장

### 문제 A: AI가 파일을 너무 많이 건드림
요청 문장:
- "수정 범위를 `app/api/toss`와 `tests/api`로 제한해. 다른 파일 건드리지 마."

### 문제 B: 코드는 됐는데 설명이 없음
요청 문장:
- "왜 이 변경이 필요한지 5줄 요약과 리스크 3개를 같이 보고해."

### 문제 C: 테스트 없이 끝냈다고 함
요청 문장:
- "명령 실행 로그 기반으로 `test/lint/build` 결과를 반드시 포함해."

### 문제 D: 요구사항을 잘못 이해함
요청 문장:
- "수정 전에 내가 이해한 요구사항을 3개 항목으로 재진술해."

## 8. Claude Code + Cursor 혼합 사용법

### 권장 역할 분리
- Cursor: 기능 구현/수정, 파일 기반 작업
- Claude Code: 큰 단위 리팩터링, 문서화/기획 정리, 반복 검증 루프

### 실제 권장 흐름
1. Cursor에서 티켓 구현
2. Claude Code로 코드리뷰 관점 점검
3. Cursor에서 수정 반영
4. Claude Code로 최종 테스트/문서 정리

## 11. 작업 요청 템플릿 모음 (복붙용)

### 템플릿 A: 기능 추가
```md
티켓: BILL-005
목표: 결제 실패 사용자에게 결제수단 변경 CTA 추가
범위: app/payments, components
제약: 기존 성공/실패 콜백 라우트 유지
DoD:
- 실패 사용자에게만 CTA 노출
- 성공 사용자/비로그인 사용자 미노출
검증:
- npm run test
- npm run lint
- npm run build
결과 보고:
- 변경 파일
- 테스트 결과
- 남은 리스크
```

### 템플릿 B: 버그 수정
```md
버그: /payments/subscription/success에서 특정 케이스 500 발생
재현 단계:
1) ...
2) ...
기대 결과: 500 대신 사용자 안내 메시지와 재시도 버튼 표시
범위: app/payments/subscription/success, app/api/toss/*
검증: 재현 케이스 기준 테스트 추가 후 통과
```

### 템플릿 C: 문서/운영 정리
```md
작업: WH-003, WH-004 관련 운영 Runbook 문서화
산출물:
- 실패 탐지 방법
- 수동 재처리 절차
- 장애 공지 템플릿
형식: docs/*.md
```

## 12. 마지막 원칙
- 빠르게 만들기보다, "검증 가능한 상태"로 만들기
- 큰 요청 1번보다, 작은 요청 5번이 안전함
- 항상 로그를 남기면 다음 작업이 쉬워진다
